## Implementation Details

Create a `fetchLocations()` method that loads all park locations from the Drupal endpoint, and add the `ParkLocation` type. The locations data is shared between the BrickFilter location list (this phase) and the LocationExplorer overlay (Phase 5).

**Current state:** No location data is fetched at the app level. `BrickCard.vue` fetches individual park locations per-brick on mount, but there is no shared list of all locations.

**API call:**

```
GET {apiUrl}parkLocations?fields[parkLocation]=name&include=brick_zone_image&sort=name
```

Returns all park locations sorted alphabetically with their associated zone map images.

**New type — add to `src/types/index.ts`:**

```typescript
export interface ParkLocation {
  id: string
  name: string
  mapImageUrl: string   // from included brick_zone_image → image_style_uri.full_img
}
```

Also add a response type for the locations endpoint:

```typescript
export interface ParkLocationsApiResponse {
  data: Array<{
    id: string
    attributes: { name: string }
    relationships: {
      brick_zone_image?: { data: { id: string } | null }
    }
  }>
  included?: Array<{
    id: string
    attributes: { image_style_uri?: { full_img?: string } }
  }>
}
```

**Where to place `fetchLocations()`:**

`App.vue` — call on mount, store as `locations: ParkLocation[]` in data. Pass down to both `BrickFilter` (via prop) and later to `LocationExplorer` (Phase 5).

```typescript
// In App.vue
data() {
  return {
    showLocationExplorer: false,
    inscription: '',
    locationIds: [] as string[],
    locations: [] as ParkLocation[],
  }
},
methods: {
  async fetchLocations() {
    try {
      const url = this.apiUrl +
        'parkLocations?fields[parkLocation]=name&include=brick_zone_image&sort=name';
      const response = await axios.get<ParkLocationsApiResponse>(url);
      this.locations = response.data.data.map((loc) => {
        // Find the matching included image by relationship ID
        const imageRelId = loc.relationships.brick_zone_image?.data?.id;
        const included = response.data.included?.find((inc) => inc.id === imageRelId);
        return {
          id: loc.id,
          name: loc.attributes.name,
          mapImageUrl: included?.attributes?.image_style_uri?.full_img ?? '',
        };
      });
    } catch {
      this.locations = [];
    }
  },
},
mounted() {
  this.fetchLocations();
},
```

**Note:** `App.vue` currently does not inject `defaultUrl`. It needs access to the API base URL. Options:

- Import the injection keys and add an `inject` block to App.vue (mirrors TheBricks/BrickCard pattern).
- Or read `import.meta.env.DEV_DRUPAL_ENDPOINT` directly (since App.vue is the provider, it can read the env var without inject).

The simpler approach is reading the env var directly, since `App.vue` is the component that provides it in the first place.

## Findings

- `App.vue` now owns `locations`, `locationIds`, and `totalCount` so the location filter and later location explorer can share a single park-location dataset.
- `fetchLocations()` uses the existing injected Drupal JSON:API base URL, avoiding a second config path in the app shell.
- Live API verification showed the relationship/include name is `brick_zone_image`, not `field_brick_zone_image`; using the original field name returns a 400 from the current endpoint.
- Verified after implementation with `npm run lint` and `npm run typecheck`.
