# Task 6.5 — Finalize BrickCard Accessibility Contract and Bottom CTA Behavior

**Phase:** 6 — BrickCard Enhancements
**Spec:** [§3.9 BrickCard Hover, Focus & Keyboard Interaction](../03-features.md#39-brickcard-hover-focus--keyboard-interaction)
**Depends on:** Tasks 6.1–6.4

---

## Implementation Details

Finalize the BrickCard interaction contract so the mouse and keyboard paths match the wireframe without redundant tab stops.

**Scope of this task:**

1. Ensure the bottom CTA remains mouse-clickable but is removed from the tab order.
2. Align accessible labels across the two distinct actions.
3. Confirm the "Coming Soon" branch suppresses enlarge affordances but still exposes location-details access.
4. Align the location-details map flow with shared Drupal image styles instead of raw file URLs.

### Bottom CTA behavior

The bottom action remains visible as a conventional button for mouse/touch users, but it should not create a third tab stop after Tasks 6.3–6.4.

Expected markup direction:

```html
<button
  class="..."
  tabindex="-1"
  @click.stop="openMap"
>
  View Location Details
</button>
```

### Accessibility contract

- Card wrapper:
  - `role="button"`
  - `aria-label="View location details for {inscription}"`
- Enlarge button:
  - `aria-label="Enlarge brick image"`
- Bottom CTA:
  - visually present
  - skipped in keyboard tab order via `tabindex="-1"`

### Event separation

- Bottom CTA click should open location details without also triggering unexpected wrapper behavior.
- Enlarge button click/keydown remains isolated from the wrapper action.
- Image-area click opens enlarge only for non-placeholder bricks.
- "Coming Soon" cards still expose only the wrapper-level keyboard location-details action plus the visible bottom location button for mouse/touch users.

### Shared location-map styling

Since the BrickCard map modal now prefers shared location data when available, update the shared park-location bootstrap flow to preserve Drupal image styles rather than only storing raw file URLs.

Expected direction:

- `App.vue` / shared location bootstrap requests `fields[file--file]=uri,url,image_style_uri`
- `ParkLocation.mapImageUrl` resolves from `image_style_uri.brick_large` (or the agreed large-map style), with raw file URL only as fallback
- BrickCard location-details modal reuses that styled shared map URL whenever possible

This keeps the modal map behavior aligned with the rest of the Drupal styled-image pipeline and avoids loading oversized originals when a style is already available.

### Files to modify

- `src/types/index.ts`
- `src/components/__tests__/App.spec.ts` (if the shared location fetch shape changes)
