## Implementation Details

Add the second keyboard tab stop: a dedicated "ENLARGE BRICK" button that opens the full-size image modal. This task also makes the real-image media area clickable for mouse users while ensuring the location-details and enlarge actions do not conflict.

**Current state:** Clicking the image already opens the image modal, but there is no explicit keyboard target for that action and no coordinated hover/focus visibility for the enlarge affordance.

### Enlarge button behavior

Render a real `<button>` in the image area for non-placeholder bricks:

```html
<button
  v-if="!isComingSoon"
  class="
    tw-absolute tw-top-2 tw-right-2
    tw-bg-black/70 tw-text-white tw-font-oswald tw-uppercase
    tw-px-3 tw-py-1 tw-text-sm
    tw-opacity-0 group-hover:tw-opacity-100 group-focus-within:tw-opacity-100
    focus:tw-opacity-100
    tw-transition-opacity tw-duration-200
  "
  aria-label="Enlarge brick image"
  @click.stop="openImg"
  @keydown.enter.stop.prevent="openImg"
  @keydown.space.stop.prevent="openImg"
>
  Enlarge Brick
</button>
```

### Interaction rules

- The button is the **second tab stop** after the card wrapper.
- `Enter` / `Space` on the button open the image modal.
- The image area itself remains clickable for mouse users and also opens the image modal, but only when `!isComingSoon`.
- The button becomes visible on hover and on `focus-within`, then remains visible when it itself is focused.
- Event handlers on the button must not leak into the bottom CTA or wrapper-level keyboard action path.

### Coming Soon variant

- The enlarge button is **not rendered** for `isComingSoon` bricks.
- The image area does **not** open the image modal for `isComingSoon` bricks.
- "Coming Soon" bricks therefore have only one tab stop: the wrapper/card action.
