## Implementation Details

### A. SearchStax Service Tests

**File:** `src/services/__tests__/searchstax.spec.ts`

Test the `searchBricks` function with mocked axios:

**Test cases:**

1. **Keyword-only search** — Verify the emselect URL is built correctly:
   - `q=*:*`
   - `fq=tcngramm_X3b_en_description:{keyword}`
   - `rows` and `start` match params
   - `fl=ss_uuid,ss_zone_uuid,ss_file_img_uuid,tcngramm_X3b_en_description`
   - `Authorization: Token {token}` header is set
   - Returns `{ bricks: Brick[], numFound: N }` with correctly mapped fields

2. **Combined keyword + single location** — Verify URL includes:
   - `fq=ss_zone_uuid:({uuid})`
   - Keyword fq is also present

3. **Combined keyword + multiple locations** — Verify URL includes:
   - `fq=ss_zone_uuid:({uuid1} OR {uuid2})`

4. **Brick object mapping** — Verify SearchStax fields map correctly to Brick interface:
   - `ss_uuid` → `Brick.id`
   - `tcngramm_X3b_en_description[0]` → `Brick.inscription`
   - `ss_file_img_uuid` → `Brick.brickImage`
   - `ss_zone_uuid` → `Brick.brickParkLocation`

5. **Empty results** — Response with `numFound: 0, docs: []` returns `{ bricks: [], numFound: 0 }`.

6. **Network error** — axios throws → `searchBricks` throws (does not catch).

### B. TheBricks Routing Tests

**File:** `src/components/__tests__/TheBricks.spec.ts` (extend existing)

These tests verify that `fetchBricks()` calls the correct data path based on the combination of `inscription` and `locationIds` props.

**Setup:** Mount TheBricks with `global.provide` for all four injection keys (Drupal endpoint, Drupal origin, SearchStax endpoint, SearchStax token). Mock axios and the `searchBricks` import.

**Test cases:**

1. **Default browse (no filters)** — `fetchBricks()` calls Drupal `/bricks?sort=brickInscription&page[limit]=20`.

2. **Location-only filter** — `fetchBricks()` calls Drupal with `filter[brickParkLocation.id]`.

3. **Keyword-only search** — `fetchBricks()` calls `searchBricks()` and uses the returned `Brick[]` directly (no Drupal call for hydration).

4. **Combined keyword + location** — `fetchBricks()` calls `searchBricks()` with `locationIds` param.

5. **SearchStax failure fallback** — `searchBricks()` throws → `fetchBricks()` falls back to Drupal `CONTAINS` keyword search.

6. **Debounce behavior** — Setting `inscription` to "bru" then quickly to "bruce" fires only one fetch (after 500ms idle). Use `vi.useFakeTimers()` and `vi.advanceTimersByTime(500)`.

7. **Clear keyword fires immediately** — Setting `inscription` to "" triggers fetch with no debounce wait.

8. **Pagination with SearchStax** — `numFound` from SearchStax (not Drupal `meta.count`) determines `totalPages`.
