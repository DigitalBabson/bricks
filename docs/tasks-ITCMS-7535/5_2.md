## Implementation Details

Connect the existing `LocationExplorerTrigger` components (desktop hero placement + floating mobile placement) to open/close the new `LocationExplorer` overlay via `App.vue` state. Also reposition the desktop trigger to sit flush against the viewport edge on most screens, anchoring to the content column on very wide viewports (≥ 2048px).

**Current state (after Phase 4):** `App.vue` already has:

- `showLocationExplorer: false` in data.
- `locations: ParkLocation[]` populated by `fetchLocations()` on mount.
- `<location-explorer-trigger>` in two placements:
  - Inside `<app-hero>` with `@openLocations="showLocationExplorer = true"` (desktop).
  - Below `</main>` with `floating: true` and `@openLocations="showLocationExplorer = true"` (mobile).

What's **missing** is the `<location-explorer>` component itself in the template and its import. The desktop trigger also needs to be repositioned (currently inside the `max-w-brickMWL` content container — should be outside it).

**Do not add a second fetch path for the overlay.** Opening the overlay should reuse the already-fetched `locations` prop from `App.vue`. Clicking a location in the overlay should only update the selected ID and rendered `<img src>`; it should not trigger a follow-up JSON:API request for that zone unless the shared bootstrap fetch design changes later.

**Endpoint scope:** `App.vue` may call `{DEV_DRUPAL_ENDPOINT}/parkLocations?include=field_brick_zone_image,field_brick_zone_image.field_media_image&fields[parkLocation]=name,field_brick_zone_image&fields[media--image]=field_media_image&fields[file--file]=uri,url` for the shared locations dataset, but the existing bricks endpoint wiring stays unchanged in this phase.

### Part A — Add LocationExplorer overlay to App.vue

1. Import `LocationExplorer`:

```typescript
import LocationExplorer from './components/LocationExplorer.vue'
```

2. Register in `components`:

```typescript
components: {
  // ... existing
  LocationExplorer,
},
```

3. Add the overlay to the template (conditionally rendered):

```html
<!-- After the closing </div> of the root wrapper, or inside it before the close -->
<location-explorer
  v-if="showLocationExplorer"
  :locations="locations"
  @close="showLocationExplorer = false"
/>
```

Since `LocationExplorer` uses `<teleport to="body">`, the exact placement in the `App.vue` template doesn't affect DOM position. Place it after `<app-footer />` for readability.

### Part B — Reposition desktop trigger in AppHero.vue

The desktop `LocationExplorerTrigger` currently sits inside the `tw-max-w-brickMWL tw-mx-auto` content container, which anchors it to the content edge. On most screens the content container fills the viewport so this looks fine, but on wide monitors the button drifts inward from the viewport edge and becomes less noticeable.

**New behavior:** The trigger should be flush to the viewport right edge on normal screens, then anchor to the content column edge on very wide screens (≥ 2048px).

**Step 1 — Add `3xl` breakpoint to `tailwind.config.js`:**

```js
// tailwind.config.js → theme.extend
screens: {
  '3xl': '2048px',
},
```

**Step 2 — Move the trigger outside the content container in `AppHero.vue`:**

```html
<section class="tw-relative tw-w-full ...">
  <!-- Full-coverage semi-transparent overlay -->
  <div class="tw-absolute tw-inset-0 tw-bg-white/80"></div>

  <!-- Desktop trigger: flush to viewport edge, anchored to content at 3xl -->
  <location-explorer-trigger
    class="
      tw-hidden md:tw-block
      tw-absolute tw-top-12 tw-z-10
      tw-right-6
      3xl:tw-right-[calc((100%-1170px)/2+24px)]
    "
    @openLocations="$emit('openLocations')"
  />

  <!-- Content container (search form) -->
  <div class="tw-relative tw-max-w-brickMWL tw-mx-auto tw-px-6 tw-py-8 md:tw-py-10 lg:tw-h-full">
    <div class="lg:tw-absolute lg:tw-left-1/2 lg:-tw-bottom-[70px] lg:tw-w-full lg:tw-max-w-[576px] lg:-tw-translate-x-1/2">
      <slot />
    </div>
  </div>
</section>
```

**How the `calc` works:**

- The hero `<section>` is `tw-relative tw-w-full`, so `100%` = viewport width.
- `(100% - 1170px) / 2` = the horizontal margin on each side of the centered content container.
- Adding `24px` (`tw-right-6` equivalent) puts the trigger at the content edge with the same padding.
- Below 2048px the content container fills most of the viewport anyway, so `tw-right-6` is effectively flush to the viewport edge.

**Verify trigger wiring:**

Both trigger placements already emit `@openLocations` which sets `showLocationExplorer = true`. The new `LocationExplorer` emits `@close` which sets it back to `false`. No changes to `LocationExplorerTrigger.vue` are needed.

**Flow:**

```
User clicks "VIEW MAP OF BRICK LOCATIONS" (either placement)
  → LocationExplorerTrigger emits openLocations
  → App.vue sets showLocationExplorer = true
  → LocationExplorer renders (teleported to body)
  → User browses locations, clicks ×, ESC, or backdrop
  → LocationExplorer emits close
  → App.vue sets showLocationExplorer = false
  → LocationExplorer unmounts
```

**Files to modify:**

- `src/App.vue` — import `LocationExplorer`, add to template with `v-if`
- `src/components/AppHero.vue` — move trigger outside content container, add `3xl:` positioning
- `tailwind.config.js` — add `screens: { '3xl': '2048px' }`
