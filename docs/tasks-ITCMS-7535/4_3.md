## Implementation Details

Add `locationIds` state to the data flow and wire it into `TheBricks.vue` so that selecting one or more park locations filters the brick results.

**Current state:** `TheBricks.vue` receives `inscription` as a prop from `App.vue` and fetches bricks via a single Drupal URL. There is no location filter parameter. In the phased plan, the inscription field remains visible in the UI, but its real data-routing behavior belongs to Phase 3/Searchstax rather than this Phase 4 Drupal work.

**Changes to `App.vue`:**

1. Add `locationIds: []` to `App.vue` data (alongside existing `inscription`).
2. Pass `locationIds` down to `TheBricks` as a prop.
3. Bind `v-model:locationIds` on `BrickFilter` (in the hero slot).
4. Pass `locations` to `BrickFilter` as a prop.
5. Handle the `clearAll` emit from BrickFilter (reset both `inscription` and `locationIds` to empty).

```html
<!-- App.vue template -->
<app-hero @openLocations="showLocationExplorer = true">
  <brick-filter
    v-model:inscription="inscription"
    v-model:locationIds="locationIds"
    :locations="locations"
    @clearAll="clearAllFilters"
  />
</app-hero>
<the-bricks
  :inscription="inscription"
  :locationIds="locationIds"
/>
```

```typescript
// App.vue methods
clearAllFilters() {
  this.inscription = '';
  this.locationIds = [];
},
```

**Changes to `TheBricks.vue`:**

1. Add `locationIds` prop: `{ type: Array as PropType<string[]>, default: () => [] }`.

2. Add a watcher on `locationIds` that resets `currentPage` to 1 and re-fetches:

   ```typescript
   watch: {
     locationIds() {
       this.currentPage = 1;
       this.fetchBricks();
     },
   },
   ```

3. Update `fetchBricks()` to add the location filter branch. Phase 4 only handles the two Drupal-backed paths (location-only and default browse). Keyword search paths (keyword-only and combined keyword + location) are owned entirely by Phase 3 and should not be implemented as temporary Drupal `CONTAINS` behavior here.

   ```typescript
   async fetchBricks() {
     try {
       const offset = (this.currentPage - 1) * this.pageSize;
       const locationIds = this.locationIds;

       let url: string;

       if (locationIds.length > 0) {
         // LOCATION ONLY → Drupal with park location IN filter
         url = this.apiUrl +
           `bricks?page[limit]=${this.pageSize}` +
           `&sort=brickInscription` +
           `&filter[brickParkLocation.id][operator]=IN` +
           `&filter[brickParkLocation.id][value]=${locationIds.join(',')}` +
           `&page[offset]=${offset}`;
       } else {
         // DEFAULT BROWSE → Drupal
         url = this.apiUrl +
           `bricks?page[limit]=${this.pageSize}` +
           `&sort=brickInscription` +
           `&page[offset]=${offset}`;
       }

       const response = await axios.get<BrickApiResponse>(url);
       // ... existing response handling ...
     } catch { /* ... */ }
   }
   ```

4. Update the empty-results message logic: show "No bricks match your criteria" when the active **location** filter set returns zero results. Keyword-driven empty states are part of Phase 3.

5. `fetchBricks()` no longer takes a `search` parameter — it reads `this.locationIds` directly from props/state. `inscription` remains App-level UI state so the field and later clear-all flow can persist into Phase 3, but it should not alter Phase 4 fetch routing.

**`totalCount` contract:**

`TheBricks` may still emit `update:totalCount` for future parent-level uses, but the current filter UI does not render a `Showing X bricks` summary in `BrickFilter`.

**TheBricks.vue:**

```typescript
emits: ['update:totalCount'],

// Inside fetchBricks(), after parsing the response:
const totalCount = response.data.meta?.count ?? 0;
this.totalPages = Math.ceil(totalCount / this.pageSize) || 1;
this.$emit('update:totalCount', totalCount);
```

- [x] `App.vue` passes `locationIds` to both `BrickFilter` (v-model) and `TheBricks` (prop).

## Findings

- `TheBricks.vue` still emits `update:totalCount`, but the current design no longer feeds that value into `BrickFilter`.
- Phase 4 routing stays scoped to Drupal-backed browse and location-only filtering. Combined keyword + location behavior remains intentionally unimplemented in the data layer.
- The inscription field remains visible in the UI, but in Phase 4 it is only carried as shared state for pills/reset behavior; it does not change the Drupal fetch path until Phase 3 integrates Searchstax.
- Temporary testing exception: keyword-only inscription search has been re-enabled against the current Drupal endpoint via `brickInscription CONTAINS` so QA can exercise the field before Phase 3 lands. This is not the final architecture for production search.
- Verified after implementation with `npm run lint` and `npm run typecheck`.
