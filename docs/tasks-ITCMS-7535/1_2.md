**Key implementation points:**

- Background image via inline style `background-image: url(...)` reading `import.meta.env.DEV_HERO_IMAGE`.
- Wrapper: `tw-relative tw-w-full tw-bg-cover tw-bg-center`, min-height ~350px desktop / ~250px mobile.
- Default `<slot>` for `BrickFilter` to be rendered inside.
- Search form container: semi-transparent white backdrop, left-aligned on desktop, full-width on mobile.
- `LocationExplorerTrigger` renders as the desktop CTA: `tw-bg-brickBabsonGreen tw-text-white tw-font-oswald tw-uppercase tw-px-6 tw-py-3`, positioned top-right.
- The trigger emits `@openLocations` to parent. The overlay itself is wired in Phase 5 вҖ” for now the event just fires.

**Responsive behavior (from XD mobile wireframe):**

- **Desktop:** trigger in top-right of hero; search form overlays left/center.
- **Mobile:** trigger becomes a floating fixed-position control in the **bottom-right corner of the viewport** so it remains visible while scrolling. Render the trigger in both locations; toggle visibility with `tw-hidden md:tw-block` (hero) and `md:tw-hidden` (mobile floating instance). The mobile instance will be placed in `App.vue` layout (see Task 1.4).

**Emits:**

| Event | Payload | Description |
|---|---|---|
| `openLocations` | вҖ” | Triggers the location explorer overlay in App.vue |

## Findings

- Hero image URL read via computed property `heroImage` from `import.meta.env.DEV_HERO_IMAGE`, with empty-string fallback.
- Min-height uses Tailwind arbitrary values: `tw-min-h-[250px] md:tw-min-h-[350px]`.
- Slot container uses the hero overlay treatment, and the search panel is constrained to a `540px` max width on desktop.
- The desktop `LocationExplorerTrigger` uses `tw-hidden md:tw-block` for responsive visibility. The complementary mobile instance is rendered in `App.vue` with `md:tw-hidden` and fixed positioning.
- 8 unit tests in `AppHero.spec.ts`.
