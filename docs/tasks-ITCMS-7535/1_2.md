# Task 1.2 — Create `AppHero.vue`

**Phase:** 1 — Page Layout
**Spec:** [§3.7 Hero Image](../03-features.md#37-hero-image-with-search-overlay)

## Implementation Details

Create `src/components/AppHero.vue` — a full-width hero banner with a background image, a `<slot>` for the search form, and desktop placement for the reusable `LocationExplorerTrigger` component.

**Structure (desktop, from XD wireframe):**

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   [Full-width hero background image]    [VIEW MAP OF BRICK      │
│                                          LOCATIONS] button      │
│    ┌──────────────────────────────────┐                         │
│    │   <slot /> (BrickFilter)         │                         │
│    └──────────────────────────────────┘                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Key implementation points:**

- Background image via inline style `background-image: url(...)` reading `import.meta.env.DEV_HERO_IMAGE`.
- Wrapper: `tw-relative tw-w-full tw-bg-cover tw-bg-center`, fixed `355px` on mobile, ~350px minimum on tablet, and a fixed `276px` hero height on desktop.
- Default `<slot>` for `BrickFilter` to be rendered inside.
- Search form container: full-height on mobile, centered, and capped at `700px` wide across breakpoints so it does not span edge to edge. On desktop it remains centered with `48px` horizontal padding. The panel uses square corners.
- On desktop, the slotted search form intentionally overhangs the bottom edge of the hero via a CSS negative top margin while staying in normal flow, and the current layout resolves to about `160px` of space below the hero image before the grid starts.
- On mobile, the grid below the hero keeps a top gap that matches the grid’s horizontal inset (`32px` / `tw-pt-8`) so the spacing feels consistent with the card gutter.
- `LocationExplorerTrigger` renders as the desktop CTA: `tw-bg-brickBabsonGreen tw-text-white tw-font-oswald tw-uppercase tw-px-6 tw-py-3`, positioned top-right.
- The trigger emits `@openLocations` to parent. The overlay itself is wired in Phase 5 — for now the event just fires.

**Responsive behavior (from XD mobile wireframe):**

- **Desktop:** trigger in top-right of hero; search form overlays left/center.
- **Mobile:** trigger becomes a floating fixed-position control in the **bottom-right corner of the viewport** so it remains visible while scrolling. Render the trigger in both locations; toggle visibility with `tw-hidden md:tw-block` (hero) and `md:tw-hidden` (mobile floating instance). The mobile instance will be placed in `App.vue` layout (see Task 1.4).

**Emits:**

| Event | Payload | Description |
|---|---|---|
| `openLocations` | — | Triggers the location explorer overlay in App.vue |

## Findings

- Hero image URL read via computed property `heroImage` from `import.meta.env.DEV_HERO_IMAGE`, with empty-string fallback.
- Min-height uses Tailwind arbitrary values: `tw-min-h-[250px] md:tw-min-h-[350px]`.
- Slot container uses the hero overlay treatment, and the search panel is constrained to a `540px` max width on desktop.
- The desktop `LocationExplorerTrigger` uses `tw-hidden md:tw-block` for responsive visibility. The complementary mobile instance is rendered in `App.vue` with `md:tw-hidden` and fixed positioning.
- 8 unit tests in `AppHero.spec.ts`.

## Note — Wireframe Alignment (2026-03-11)

- The implemented hero treatment uses a full-coverage white overlay on top of the hero image at **0.5 opacity**.
- The search UI itself is a separate courtyard-green panel placed within the hero; it is not the white overlay surface.
- The search panel uses a **700px max width** in the implemented layout and is centered on desktop.
- On mobile, the search panel fills the hero vertically from top to bottom within the `355px` hero height and caps at **700px** wide.
- On desktop, the search panel intentionally overhangs the hero bottom edge via a CSS negative top margin while staying in normal flow, and the current layout resolves to about **160px** below the hero image before the brick grid begins.
- On mobile, the grid wrapper applies a top gap equal to the grid’s side inset so the hero/search area and the first card row are separated consistently.
