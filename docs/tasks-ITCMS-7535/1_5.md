# Task 1.5 — Move `BrickFilter` into Hero Slot

**Phase:** 1 — Page Layout
**Spec:** [§3.7 Hero Image — Implementation approach](../03-features.md#371-new-appherovue-component)
**Depends on:** Tasks 1.2, 1.4

---

## Implementation Details

Move the `<brick-filter>` rendering from inside `TheBricks.vue` into the `<app-hero>` default slot in `App.vue` (or the parent managing the hero).

**Current state:** `BrickFilter` is rendered inside `TheBricks.vue` as a sibling of the brick grid, bound via `v-model:inscription`.

**Target state:**

```html
<!-- In App.vue or TheBricks.vue parent section -->
<app-hero @openLocations="showLocationExplorer = true">
  <brick-filter
    v-model:inscription="inscription"
    v-model:locationId="locationId"
    :locations="locations"
    :resultCount="totalCount"
    @clearAll="clearAllFilters"
  />
</app-hero>
```

**Key points:**

- The `inscription` state currently lives in `TheBricks.vue`. For this task, pass it through or lift it. The simplest approach: keep state in `TheBricks` and use a scoped slot or emit pattern. Alternatively, lift `inscription` to `App.vue` and pass down.
- The `locationId`, `locations`, and `clearAll` bindings are not functional yet (those come in Phase 4). For now, wire only `v-model:inscription` and leave the other bindings as stubs or omit them.
- The visual result: the search input is overlaid on the hero image background instead of sitting above the brick grid on a plain background.
- Ensure `BrickFilter` container has its semi-transparent white backdrop styling within the hero context.
- On desktop, the filter should overhang the hero bottom edge rather than sitting fully inside the hero bounds.

**Files to modify:**

- `src/App.vue` — add BrickFilter inside AppHero slot
- `src/components/TheBricks.vue` — remove BrickFilter rendering from template

## Findings

- **State lift:** `inscription` moved from `TheBricks.data()` to `App.vue.data()`. TheBricks now receives it as a prop (`inscription: { type: String, default: '' }`). The watcher on `inscription` still triggers `fetchBricks()` in TheBricks.
- BrickFilter import and registration moved from TheBricks to App.vue.
- The `v-model:inscription` binding flows: App.vue ↔ BrickFilter (two-way), App.vue → TheBricks (one-way prop).

## Note — Wireframe Alignment (2026-03-11)

- The visual backdrop behind the hero content comes from `AppHero` as a white overlay at **0.5 opacity** across the full hero image.
- `BrickFilter` should continue to read as its own courtyard-green panel inside that hero treatment rather than as a semi-transparent white card.
- Search behavior has since been standardized at **3+ characters with a 500ms debounce**; the older references to a 4-character threshold in this task should be treated as superseded by the current feature docs.
- Desktop layout now treats the filter as an overhanging element: the hero itself is `276px` tall on desktop and reserves about `140px` below before the grid begins.
- Mobile layout keeps a simpler flow: the hero ends, then the grid wrapper adds a top gap equal to the grid’s side inset before the first row of cards.
