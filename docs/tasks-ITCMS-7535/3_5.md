# Task 3.5 — Add Drupal CONTAINS Fallback for SearchStax Failures

**Phase:** 3 — SearchStax Integration
**Spec:** [§7 Open Question #7](../07-open-questions.md) — "Fall back to Drupal CONTAINS search if Searchstax is unavailable"
**Depends on:** Task 3.4

---

## Implementation Details

Wrap the SearchStax path in `fetchBricks()` with a try/catch that falls back to the existing Drupal `CONTAINS` filter when SearchStax is unavailable (network error, timeout, 401, 500, etc.).

**Current state (after Task 3.4):** `fetchBricks()` routes keyword searches to `fetchViaSearchstax()`. If that throws, the outer catch sets `bricks = []` and shows the empty state. There's no fallback to Drupal.

**Approach:**

Update `fetchBricks()` so that when the SearchStax path fails, it attempts the Drupal `CONTAINS` search before giving up:

```typescript
async fetchBricks() {
  try {
    const hasKeyword = this.inscription.length >= 3;

    if (hasKeyword) {
      try {
        await this.fetchViaSearchstax();
        return; // SearchStax succeeded, we're done
      } catch {
        // SearchStax failed — fall back to Drupal CONTAINS below
        console.warn('SearchStax unavailable, falling back to Drupal keyword search');
      }

      // Drupal CONTAINS fallback for keyword search
      await this.fetchViaDrupalKeyword();
      return;
    }

    // Non-keyword paths (default browse, location-only) — always Drupal
    const offset = (this.currentPage - 1) * this.pageSize;
    const url = this.buildUrl(offset);
    const response = await axios.get<BrickApiResponse>(url);
    // ... existing parsing logic ...

  } catch {
    // Total failure (even Drupal is down)
    this.bricks = [];
    this.totalPages = 1;
    this.showMessage = this.locationIds.length > 0 || this.inscription.length >= 3;
    this.$emit('update:totalCount', 0);
  }
},
```

**Add `fetchViaDrupalKeyword()` method:**

This preserves the old Drupal CONTAINS logic that is being removed from `buildUrl()` in Task 3.4:

```typescript
async fetchViaDrupalKeyword() {
  const offset = (this.currentPage - 1) * this.pageSize;

  // Build Drupal CONTAINS URL (same as pre-Phase-3 keyword search)
  let url = this.apiUrl +
    `bricks?page[limit]=${this.pageSize}` +
    `&filter[brickInscription][operator]=CONTAINS` +
    `&filter[brickInscription][value]=${encodeURIComponent(this.inscription)}` +
    `&page[offset]=${offset}` +
    `&sort=brickInscription`;

  // If locations are also selected, add location filter
  if (this.locationIds.length > 0) {
    const encodedIds = this.locationIds.map((id) => encodeURIComponent(id)).join(',');
    url += `&filter[brickParkLocation.id][operator]=IN` +
           `&filter[brickParkLocation.id][value]=${encodedIds}`;
  }

  const response = await axios.get<BrickApiResponse>(url);

  const data: Brick[] = response.data.data.map((bricks) => {
    const brickImage: string = bricks.relationships.brickImage.data == null
      ? 'default'
      : bricks.relationships.brickImage.data.id;
    return {
      id: bricks.id,
      inscription: bricks.attributes.brickInscription,
      brickImage,
      brickParkLocation: bricks.relationships.brickParkLocation.data?.id ?? '',
    };
  });

  const hasActiveFilter = this.locationIds.length > 0 || this.inscription.length >= 3;
  this.showMessage = hasActiveFilter && data.length === 0;
  this.bricks = data;

  const totalCount = response.data.meta?.count ?? 0;
  this.totalPages = Math.ceil(totalCount / this.pageSize) || 1;
  this.$emit('update:totalCount', totalCount);
},
```

**Design notes:**

- The fallback is silent to the user — they get results from Drupal without knowing SearchStax failed. The `console.warn` helps developers diagnose issues.
- The Drupal CONTAINS fallback also supports combined keyword + location filtering by appending the `brickParkLocation.id` filter. This means all four search modes work even when SearchStax is down.
- No retry logic — if SearchStax fails once, we fall back immediately. Retries would add latency on an already-failed path.
- Consider extracting the Drupal brick-parsing logic into a helper to reduce duplication between `fetchViaDrupalKeyword()` and the default Drupal path. This is optional and can be done as a cleanup pass. (Note: `fetchViaSearchstax()` no longer parses Drupal responses — it uses `Brick[]` directly from the service.)
